# Star-DART 项目文档模板

## 文档目标

这份文档不是 README 翻译，而是帮助未来的自己快速判断：

- 这个项目解决什么问题？
- 为什么当时值得 Star？
- 它能不能被复用到自己的 Agent、Skill、内容、产品或研发流程里？
- 如果要上手，最快路径是什么？

## 标准结构

````markdown
# {中文标题} - {owner/repo}

| 字段 | 内容 |
| --- | --- |
| 文档标题 | {中文标题} - {owner/repo} |
| 仓库 | {owner/repo} |
| GitHub | https://github.com/{owner}/{repo} |
| 技术方向 | {技术方向} |
| 主要语言 | {语言，如未知写“待确认”} |
| Stars / Forks | {stars} / {forks}，如未知写“待确认” |
| License | {license，如未知写“待确认”} |
| Star 时间 | {starred_at，如未知写“待确认”} |
| 生成时间 | {YYYY-MM-DD} |

## 🎯 项目定位

用 2 段说明：这是一个什么项目，面向谁，解决什么问题，以及为什么适合进入 Star-DART。

一句话判断：用一句话说清它未来最可能怎样被复用。

## 🏗️ 技术结构与实现思路

### 架构图

如果项目存在明显模块关系、调用链路、分层结构、数据流或工具链，必须在本节开头插入 1 张架构图。不要用普通 Markdown 代码块写 ```mermaid；飞书文档应使用画板块：

```xml
<whiteboard type="mermaid">
graph LR
  A[输入/用户] --> B[核心模块]
  B --> C[输出/交付物]
</whiteboard>
```

图表选择规则：

- 简单链路、思维导图、时序图、类图、饼图、甘特图：可直接用 `<whiteboard type="mermaid">`。
- 分层系统架构、模块框图、复杂流程、需要更好视觉层级的图：优先用 `<whiteboard type="svg">完整自包含 SVG</whiteboard>`，或先插入空白画板后用 `lark-whiteboard` 的 DSL / SVG 路径写入。
- 架构图只表达关键模块和关系，避免把 README 目录或所有函数都塞进图里。

## 推荐篇幅

- 不要写成 8 行简版，也不要把 README 全文搬进来。
- 正文目标是 3 到 5 分钟能看完：通常 1200 到 2200 个中文字。
- 每个二级章节都要有实质内容；不要只写一句“待确认”。
- 不要在自动生成文档末尾附加长篇 README 摘录、源码全文或“本地详稿摘录”。如果资料很长，只提炼到上面的结构里。
- 技术结构章节默认应有 1 张架构图；只有资料极少、确实无法判断模块关系时，才写“架构图待补充：资料不足，需要阅读源码/运行后确认”。

## 写作要求

- 不要逐句翻译 README。
- 不要把项目描述写成广告语。
- 不要编造 README 或公开资料中没有的信息。
- 优先解释“为什么值得 Star”和“未来怎么复用”。
- 技术术语、函数名、变量名、包名保留英文。
- 如果项目资料不足，明确写入“待确认”，不要硬凑。
- GitHub API、README、Release、docs 能拿到的信息要优先核验，避免只根据仓库 description 写短文。
- 开头不要使用 YAML frontmatter；项目信息必须整理成表格。
- 飞书中的 Mermaid/架构图必须通过 `<whiteboard>` 插入；不要把 ```mermaid 代码块当作最终图表。
