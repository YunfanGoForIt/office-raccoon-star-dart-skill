# 1. 场景洞察 — 什么项目？解决谁的什么问题？亮点在哪？

## 第 1 页｜项目一句话

**标题：Star-DART OPC：把 GitHub Star 变成开源项目知识资产**

**页面结论：** Star-DART 不是收藏夹，而是一个由办公小浣熊驱动的开源项目理解、入库和复盘工作流。

**核心信息：**

- 面向对象：AI 工具重度用户、开发者、学生科研用户、技术团队。
- 触发动作：用户像平常一样 Star 一个 GitHub 仓库。
- 系统结果：小浣熊定时发现新增 Star，生成项目文档，写入飞书目录和多维表格，并每周生成 PPT 回顾。
- 一句话口号：**现在看懂，未来找回，定期复盘。**

**建议画面：**

- 左侧：GitHub Star 图标和“轻动作”。
- 右侧：飞书文档、Base 仪表盘、周回顾 PPT 三个产物卡片。
- 底部流程：`Star → 文档 → 台账 → 仪表盘 → 周 PPT`。

**口播提示：**  
“Star-DART 解决的不是如何多收藏，而是收藏之后如何真正被理解、找回和复用。”

---

## 第 2 页｜真实痛点

**标题：Star 越多，复用越少**

**页面结论：** 开发者收藏了很多好项目，但项目真正开始时，仍然经常从零搜索、从零判断、从零造轮子。

**痛点拆解：**

| 阶段 | 用户动作 | 真实问题 |
| --- | --- | --- |
| 看到项目 | 点 Star | 只保存链接，没有保存判断 |
| 几天后 | 想复用 | 忘记项目名、亮点和边界 |
| 做项目时 | 让 AI 开工 | AI 可能绕过已收藏的优秀实现 |
| 团队协作 | 分享链接 | 链接无法直接变成技术选型依据 |

**目标用户：**

- 个人开发者：需要把零散开源项目变成自己的技术雷达。
- AI 编程用户：希望 Codex / Claude Code 开工前能先参考自己收藏过的项目。
- 学生和科研用户：需要把工具、论文代码和教程沉淀成长期知识库。
- 小团队：需要轻量维护“我们关注的开源项目库”。

**建议画面：**

- 一张“收藏堆积 → 记忆断层 → 重新搜索”的三段图。
- 用“链接不是资产，理解才是资产”做页面金句。

---

## 第 3 页｜产品亮点

**标题：一个 Star，自动进入一条知识生产链**

**页面结论：** Star-DART 的亮点在于把“低成本收藏动作”转化成“高价值办公交付物”。

**三层转化：**

| 原始状态 | Star-DART 后 |
| --- | --- |
| GitHub 收藏链接 | 一份结构化项目文档 |
| 个人短期记忆 | 飞书多维表格资产台账 |
| 临时阅读 | 仪表盘和每周 PPT 回顾 |

**产品亮点：**

- 低打扰：用户仍然只需要点 Star。
- 可沉淀：每个仓库都有独立飞书项目文档。
- 可管理：Base 里有分类、推荐动作、复用等级、文档链接。
- 可复盘：每周 PPT 总结新增 Star，强化记忆。

# 2. 技术方案 — 如何基于办公小浣熊实现？有什么技术亮点？实现思路和过程是怎么样的？

## 第 4 页｜整体架构

**标题：基于办公小浣熊的 Star 入库闭环**

**页面结论：** 办公小浣熊承担“计划执行 + 资料研判 + 飞书办公交付”的核心角色。

**架构流程：**

```text
办公小浣熊定时任务
  ↓
python3 scripts/check_new_stars.py
  ↓
GitHub Star API
  ↓
飞书 Base 去重
  ↓
新增仓库清单
  ↓
办公小浣熊生成项目文档
  ↓
lark-cli 写入飞书文档 / Base / 仪表盘 / Slides
```

**模块分工：**

| 模块 | 职责 |
| --- | --- |
| 定时任务 | 周期运行，不依赖系统级服务 |
| 轮询脚本 | 抓取 GitHub Star，并与飞书 Base 比对 |
| Skill 指令 | 约束小浣熊如何生成文档、入库和更新目录 |
| 飞书文档 | 存放每个仓库的项目档案 |
| 飞书 Base | 作为资产台账和去重事实来源 |
| 飞书仪表盘 | 统计分类、动作和复用等级 |
| 飞书 Slides / PPT | 每周回顾与记忆强化 |

---

## 第 5 页｜技术亮点

**标题：不是“摘要脚本”，而是办公自动化 Skill**

**页面结论：** 技术亮点是把可重复、易出错的办公链路封装成小浣熊可执行的 Skill。

**亮点 1：Base 作为去重事实来源**  改为“文档和多维表格借助飞书云文档，全部线上保存，多端随时可查”的意思

- 不用本地状态文件判断是否处理过。
- 通过仓库名称和 GitHub URL 与飞书 Base 已有记录比对。
- 保证“已经入库的仓库”不会重复生成文档。

**亮点 2：脚本和 Agent 分工清晰**

- 脚本只负责发现新增仓库和输出记录种子。
- 小浣熊负责理解项目、生成文档、操作飞书。
- 这样既稳定，又保留 Agent 的判断能力。

## 第 6 页｜实现过程

**标题：从吃灰的GitHub Star列表，沉淀到飞书知识库资产闭环**

**实现步骤：**

| 步骤 | 当前做法 |
| --- | --- |
| 1. 触发机制 | 办公小浣熊定时执行脚本，默认 10 分钟一次 |
| 2. 数据抓取 | GitHub Star API 获取最近收藏 |
| 3. 去重判断 | 读取飞书 Base 中已有仓库名称和 URL |
| 4. 项目理解 | 小浣熊按 `references/doc_template.md` 生成项目文档 |
| 5. 资产入库 | 文档链接、分类、推荐动作、复用等级写入 Base |
| 6. 看板统计 | 使用飞书 Base 仪表盘展示分布 |
| 7. 周度复盘 | 每周生成 PPT 并追加到同一飞书目录 |

## 第 8 页｜效率价值

**标题：把重复整理动作交给小浣熊，把判断留给用户复核**

**页面结论：** Star-DART 的效率价值来自减少重复阅读、重复建档、重复入库和重复复盘。

**效率提升点：**

| 原手工动作 | Star-DART 后 |
| --- | --- |
| 打开 GitHub Star 列表逐个检查 | 定时脚本只输出新增仓库 |
| 重新阅读 README 后写零散笔记 | 小浣熊按统一模板生成项目档案 |
| 手动维护 Excel | 飞书 Base 直接作为台账 |
| 手动做统计图 | 飞书 Base 仪表盘持续统计 |
| 事后忘记本周看过什么 | 每周 PPT 自动形成回顾材料 |

**数据口径建议：**

- 不伪造用户数和营收。
- 复赛演示中可以展示真实仓库文件、测试结果、飞书 Base、飞书仪表盘和预录视频。
- 如果现场要讲效率，用“同一仓库手工整理耗时 vs 小浣熊辅助整理耗时”的计时对比，不把估算包装成已验证数据。

**建议画面：**

- 左右对比：手工整理链路 vs 小浣熊自动办公链路。
- 底部强调：“当前阶段验证的是闭环效率，不是商业收入。”

## 第 9 页｜潜在用户与市场空间

**标题：从个人开源收藏，延展到团队技术雷达**

**页面结论：** 这个场景具有明确潜在用户：越依赖开源和 AI 编程的人，越需要把收藏变成可复用资产。

**潜在用户：**

| 用户 | 使用动机 |
| --- | --- |
| AI 编程重度用户 | 开工前先找自己收藏过的可复用项目 |
| 独立开发者 | 把工具、模板、框架沉淀成个人技术雷达 |
| 学生 / 科研新人 | 管理论文代码、课程项目和实验工具 |
| 技术团队 | 维护团队关注的开源项目池和选型依据 |
| 内容创作者 | 把每周技术发现整理成选题和 PPT |

**未来可延展方向：**

- 团队版：多人 Star 汇聚、统一分类、项目选型报告。
- AI 编程前置检索：在 Codex / Claude Code 开工前查询已收藏项目。
- 云服务：为不想维护 token 和定时任务的用户提供托管。
- MCP / API：把结构化开源项目知识库接入研发流程。


展开为两页！把未来部分重点讲。附上拓展思路原文：

我现在需要你进一步优化这个startart，因为我现在这次参加的是OPC能力大赛，它是要要求业务闭环和商业价值的。

所以我们不能仅仅是做一个GitHub仓库、GitHub star 仓库的整理和这个。知识库的构建，这个东西它就是一个个人使用的，所以我们想了一下，可以继续往后去拓展。

具体的拓展方向如下：
1. 提供云端订阅服务
  对于这些不方便部署的，可能没有自己的一个服务器或者Agent的，可以提供云端订阅服务。我们的star dart 云服务来去帮你处理你的GitHub新增star仓库，然后处理完成后通过邮件、飞书等等的方式推送给你，然后在云端构建你的专属项目知识库，并支持同步到CDN和飞书。
2. 构建MCP和Skills
  更进一步的可以做一个MCP和一个Skills，让你的 Claude Code 或者 Codex 每次在开始构建一个项目的时候，先通过你的这个专属知识库进行搜索。尽量减少重复造轮子，借鉴优秀的开源项目去构建。
3. 构建统一的优质开源项目知识库
  这个服务的用户多了起来之后，可以打通所有用户的项目知识库。做一个统一的，大家右脚投票出来的优质开源项目知识库，解决GitHub搜索极其困难，模糊搜索非常差，然后AI搜索开源项目。良莠不齐的情况，为大家构建项目的时候，提供高质量的开源项目查找和参考的MCP工具。

这样的话就想象力更大，商业闭环也更加的完整

## 第 11 页｜现场演示路径

**标题：现场只演示最稳定、最能证明闭环的部分**

**页面结论：** 现场演示优先保证稳定：先 dry-run，再展示飞书已有结果。

**推荐现场顺序：**

1. 打开 GitHub 仓库结构，展示 `SKILL.md`、`scripts/`、`references/`、`tests/`。
2. 运行样例命令：

   ```bash
   python3 scripts/check_new_stars.py \
     --sample examples/sample_starred_repo.json \
     --existing-json examples/base_records_empty.json
   ```

3. 展示输出中的新增仓库清单和 Base 记录种子。
4. 打开飞书目录页，展示项目子文档。
5. 打开飞书 Base，展示资产台账字段。
6. 打开飞书仪表盘，展示分类分布、推荐动作分布、复用等级分布。
7. 打开周回顾 PPT，展示“即时入库 + 每周复盘”的双节奏。

**演示重点：**

- 小浣熊不是单次问答，而是在执行一个可复用 Skill。
- 脚本只负责确定新增仓库，避免过度自动化造成不可控。
- 飞书是长期知识资产的承载层。

- 不能在 PPT 或 README 中暴露真实 GitHub 用户名、飞书链接、Base token 或 table id。
