# 1. 场景洞察 — 什么项目？解决谁的什么问题？亮点在哪？

## 第 1 页｜项目一句话

**标题：Star-DART OPC：把 GitHub Star 变成开源项目知识资产**

**页面结论：** Star-DART 不是收藏夹，而是一个由办公小浣熊驱动的开源项目理解、入库和复盘工作流。

**核心信息：**

- 面向对象：AI 工具重度用户、开发者、学生科研用户、技术团队。
- 触发动作：用户像平常一样 Star 一个 GitHub 仓库。
- 系统结果：办公小浣熊定时发现新增 Star，生成项目文档，写入飞书目录页下的多维表格，并每周生成 PPT 回顾。
- 一句话口号：**现在看懂，未来找回，定期复盘。**

**页面呈现：** `Star → 文档 → 台账 → 仪表盘 → 周 PPT`

---

## 第 2 页｜真实痛点

**标题：Star 越多，复用越少**

**页面结论：** 开发者收藏了很多好项目，但项目真正开始时，仍然经常从零搜索、从零判断、从零造轮子。

**痛点拆解：**

| 阶段 | 用户动作 | 真实问题 |
| --- | --- | --- |
| 看到项目 | 点 Star | 只保存链接，没有保存判断 |
| 几天后 | 想复用 | 忘记项目名、亮点和适用范围 |
| 做项目时 | 让 AI 开工 | AI 可能绕过已收藏的优秀实现 |
| 团队协作 | 分享链接 | 链接无法直接变成技术选型依据 |

**目标用户：**

- 个人开发者：需要把零散开源项目变成自己的技术雷达。
- AI 编程用户：希望 Codex / Claude Code 开工前能先参考自己收藏过的项目。
- 学生和科研用户：需要把工具、论文代码和教程沉淀成长期知识库。
- 小团队：需要轻量维护“我们关注的开源项目库”。

**页面金句：** 链接不是资产，理解才是资产。

---

## 第 3 页｜产品亮点

**标题：一个 Star，自动进入一条知识生产链**

**页面结论：** Star-DART 的亮点在于把“低成本收藏动作”转化成“高价值办公交付物”。

**三层转化：**

| 原始状态 | Star-DART 后 |
| --- | --- |
| GitHub 收藏链接 | 一份结构化项目文档 |
| 个人短期记忆 | 飞书多维表格资产台账 |
| 临时阅读 | 仪表盘和每周 PPT 回顾 |

**产品亮点：**

- 低打扰：用户仍然只需要点 Star。
- 可沉淀：每个仓库都有独立飞书项目文档。
- 可管理：Base 里有分类、推荐动作、复用等级、文档链接。
- 可复盘：每周 PPT 总结新增 Star，强化记忆。

# 2. 技术方案 — 如何基于办公小浣熊实现？有什么技术亮点？实现思路和过程是怎么样的？

## 第 4 页｜整体架构

**标题：基于办公小浣熊的 Star 入库闭环**

**页面结论：** 办公小浣熊承担“计划执行 + 本地命令 + 资料研判 + 飞书办公交付”的核心角色。

**架构流程：**

```text
办公小浣熊定时任务
  ↓
python3 scripts/check_new_stars.py
  ↓
GitHub Star API
  ↓
飞书 Base 去重
  ↓
新增仓库清单
  ↓
办公小浣熊生成项目文档
  ↓
lark-cli 写入飞书文档 / 目录页 / Base / 仪表盘 / Slides
```

**模块分工：**

| 模块 | 职责 |
| --- | --- |
| 办公小浣熊定时任务 | 周期运行脚本，不依赖系统级服务 |
| 本地 Python 脚本 | 抓取 GitHub Star，并与飞书 Base 比对 |
| Skill 指令 | 约束小浣熊如何生成文档、入库和更新目录 |
| 飞书文档 | 存放每个仓库的项目档案 |
| 飞书 Base 子文档 | 作为资产台账和去重事实来源 |
| 飞书 Base 仪表盘 | 统计分类、动作和复用等级 |
| 飞书 Slides / PPT | 每周回顾与记忆强化 |

---

## 第 5 页｜技术亮点

**标题：不是“摘要脚本”，而是办公自动化 Skill**

**页面结论：** 技术亮点是把可重复、易出错的办公链路封装成小浣熊可执行的 Skill。

**亮点 1：飞书云文档承载知识资产**

- 项目文档、多维表格和 PPT 都沉淀在飞书云文档体系内。
- Base 是飞书目录页下的多维表格子文档，不是散落的本地 Excel。
- 文档和表格全部线上保存，多端随时可查，也便于后续团队协作。

**亮点 2：办公小浣熊能力组合，而非单点脚本**

- 定时任务能力：按周期执行 GitHub Star 入库，不需要系统级服务。
- 本地命令能力：运行 Python 脚本，稳定完成抓取和去重。
- 文档生成能力：按 `references/doc_template.md` 为每个仓库生成项目档案。
- 飞书插件连接能力：通过 `lark-cli` 创建文档、写 Base、维护目录和仪表盘。
- PPT 生成能力：每周汇总新增 Star，生成记忆强化型回顾 PPT。

**亮点 3：脚本和 Agent 分工清晰**

- 脚本只负责发现新增仓库和输出记录种子。
- 小浣熊负责理解项目、生成文档、操作飞书。
- 这样既保证流程稳定，又保留 Agent 对开源项目的判断能力。

---

## 第 6 页｜实现过程与当前闭环

**标题：从吃灰的 GitHub Star 列表，沉淀到飞书知识库资产闭环**

**页面结论：** 当前实现已经跑通一条办公小浣熊能力链：定时任务触发、本地命令抓取、文档生成、飞书插件入库、每周 PPT 复盘。

**闭环步骤：**

| 步骤 | 当前做法 |
| --- | --- |
| 1. 定时任务 | 办公小浣熊周期执行脚本，默认 3 小时；演示时可临时改为 10 分钟 |
| 2. 本地命令 | 运行 `python3 scripts/check_new_stars.py`，抓取 GitHub Star 并与 Base 去重 |
| 3. 文档生成 | 小浣熊按 `references/doc_template.md` 为新增仓库生成项目档案 |
| 4. 飞书插件连接 | 通过 `lark-cli` 创建飞书子文档、追加目录、写入 Base、维护仪表盘 |
| 5. PPT 生成 | 每周汇总新增 Star，生成回顾 PPT 并追加到同一飞书目录 |

**可验证产物：**

| 成果 | 证明点 |
| --- | --- |
| `SKILL.md` | 定义办公小浣熊运行流程和质量检查 |
| `scripts/check_new_stars.py` | GitHub Star 抓取与飞书 Base 去重脚本 |
| `references/doc_template.md` | 每个 Star 仓库的项目文档模板 |
| `references/weekly_ppt_template.md` | 每周回顾 PPT 模板 |
| `references/scheduled_task_prompts.md` | 两个小浣熊定时任务 Prompt |
| 飞书目录页 / Base / 仪表盘 / PPT | 当前办公资产的最终承载层 |

# 3. 当前成果 — 闭环成果、用户数、营收等数据；或者提升的效率、市场潜力、潜在用户等维度

## 第 7 页｜效率价值与潜在用户

**标题：把重复整理动作交给小浣熊，把判断留给用户复核**

**页面结论：** Star-DART 的效率价值来自减少重复阅读、重复建档、重复入库和重复复盘；潜在用户是依赖开源项目和 AI 编程的人群。

**效率提升点：**

| 原手工动作 | Star-DART 后 |
| --- | --- |
| 打开 GitHub Star 列表逐个检查 | 定时脚本只输出新增仓库 |
| 重新阅读 README 后写零散笔记 | 小浣熊按统一模板生成项目档案 |
| 手动维护 Excel | 飞书 Base 直接作为台账 |
| 手动做统计图 | 飞书 Base 仪表盘持续统计 |
| 事后忘记本周看过什么 | 每周 PPT 自动形成回顾材料 |

**潜在用户：** AI 编程重度用户、独立开发者、学生 / 科研新人、技术团队和技术内容创作者。

**成果展示：** 真实仓库文件、测试结果、飞书 Base、飞书仪表盘和录制视频。

## 第 8 页｜商业拓展一：云端订阅服务

**标题：从个人 Skill，升级为免部署的 Star-DART 云服务**

**页面结论：** 当前 Skill 验证的是最小闭环；下一步可以把“个人本地运行”升级为“云端订阅服务”，形成更清晰的商业闭环。

**为什么需要云服务：**

| 当前 Skill 模式 | 云端订阅服务 |
| --- | --- |
| 用户需要配置 GitHub / 飞书 token | 用户授权后即可使用 |
| 需要办公小浣熊或本地环境定时执行 | 云端自动处理新增 Star |
| 适合个人和复赛演示 | 适合不想部署的开发者、学生和团队 |
| 输出主要进入个人飞书目录 | 可同步到邮件、飞书、CDN 或个人知识库 |

**可设计的产品形态：**

- 个人版：订阅自己的 GitHub Star，每次新增自动推送项目档案。
- 团队版：汇总团队成员 Star，形成团队开源项目资产池。
- 知识库版：云端构建专属项目知识库，支持同步到飞书和公开 CDN。

---

## 第 9 页｜商业拓展二：MCP / Skills 与优质开源项目网络

**标题：让 AI 编程开工前，先查询用户真正收藏过的好项目**

**页面结论：** 当 Star-DART 积累到足够多项目档案后，可以从“个人知识库”进一步变成 AI 编程的项目检索基础设施。

**拓展方向：**

| 方向 | 价值 |
| --- | --- |
| Star-DART MCP | Codex / Claude Code 开始项目时，先检索用户专属开源项目库 |
| Star-DART Skills | 把项目档案转成可复用的 Agent / Skill 构建参考 |
| 统一优质项目库 | 汇总多用户 Star 行为，形成“用脚投票”的高质量开源项目索引 |
| AI 搜索增强 | 缓解 GitHub 搜索模糊、AI 推荐项目良莠不齐的问题 |

**商业想象力：**

- 面向个人：少重复造轮子，开工前先复用自己收藏过的优秀实现。
- 面向团队：沉淀团队技术选型依据，降低新人调研成本。
- 面向生态：构建高质量开源项目检索 MCP，让 AI Agent 能找到更可靠的开源参考。

**增长路径：** 个人知识库 → 团队技术雷达 → 云端订阅服务 → MCP 开源项目检索网络。

**标题：为什么 Star-DART OPC 符合复赛评价重点**
