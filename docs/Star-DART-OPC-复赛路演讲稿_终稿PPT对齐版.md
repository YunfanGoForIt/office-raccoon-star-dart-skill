# Star-DART OPC 复赛路演讲稿（终稿 PPT 对齐版）

> 来源：根据 `Star-DART OPC 复赛路演 终稿.pptx` 的 10 页终稿结构，对齐原讲稿重排。
> 调整：原讲稿第 6 页“实现过程与当前闭环”压缩并入第 5 页；原第 7 页起顺延对齐 PPT 第 6-10 页。

## 第 1 页｜封面：一句话讲清

> 建议时间：0:00-0:20

各位评委老师好，我的作品叫 Star-DART OPC。

一句话介绍：它是一个把 GitHub Star 变成开源项目知识资产的办公小浣熊 Skill。

用户只需要继续收藏仓库；后续由小浣熊主动发现新增 Star，生成项目档案，写入飞书 Base，形成仪表盘和周回顾 PPT。

核心价值是：主动触发，现在看懂，未来找回，随时复用。

## 第 2 页｜痛点：Star 越多，复用越少

> 建议时间：0:20-0:50

这个项目来自一个很真实的开发者习惯：我们会收藏很多开源项目，但真正要用的时候，还是经常重新搜索、重新判断。

问题不是没有收藏，而是 Star 只有链接，没有保存收藏原因、适用场景、技术亮点和复用建议。

对 AI 编程用户更明显。如果 Codex 或 Claude Code 不知道我收藏过哪些优秀实现，它就可能绕过已有参考。

所以 Star-DART 解决的是：收藏如何变成可复用知识资产。

## 第 3 页｜定位：从链接到资产

> 建议时间：0:50-1:15

原来的 GitHub Star 列表，本质是链接堆积，知识链路停在浏览器里。

Star-DART OPC 做的是把每个新增 Star 放进一条生产链路：理解、归档、索引、复用。

最终产物不是一段摘要，而是结构化中文项目档案、飞书知识库、多维表格、看板统计和周 PPT。

也就是把“我看过”变成“我以后能找回、能判断、能直接复用”。

## 第 4 页｜架构：三层闭环

> 建议时间：1:15-1:45

技术上可以看成三层。

第一层是 Star-DART 定时任务，主动触发 GitHub Star API，维护状态快照，找出新增仓库。

第二层是办公小浣熊综合处理，结合 Skill、研究研判和飞书插件，完成理解、生成和交付。

第三层是资产落库：飞书知识库、Base 台账、仪表盘、周报和 PPT。它不是单点摘要脚本，而是一套持续运行的知识资产系统。

## 第 5 页｜核心：小浣熊承担工作流

> 建议时间：1:45-2:15

这里的小浣熊不是被调用一次来写摘要，而是在执行完整工作流。

脚本只做确定性任务：抓取 Star、读取 Base、比对去重。

小浣熊做需要判断的部分：阅读 README、DeepWiki、Release 和 Issue，判断项目价值，并生成统一结构的中文项目档案。

然后它继续写入飞书文档、Base、仪表盘和周 PPT。这个分工让流程既稳定，又保留了 Agent 的理解能力。

## 第 6 页｜结果：省掉重复整理

> 建议时间：2:15-2:45

手工整理 Star 很碎：逐个翻列表、重读 README、写笔记、维护表格、做统计、再靠记忆复盘。

Star-DART 把这些动作交给小浣熊：新增仓库自动发现，项目档案按模板生成，资产台账进飞书 Base，统计交给仪表盘，每周形成 PPT 回顾。

用户真正要做的，是复核小浣熊的判断，决定哪些项目值得继续试跑、研究或沉淀成自己的工具。

目标用户是 AI 编程用户、独立开发者、学生科研新人、技术团队和技术内容创作者。

## 第 7 页｜商业一：云端订阅服务

> 建议时间：2:45-3:05

第一个商业方向，是把个人可运行的 Skill 升级成免部署的云端订阅服务。

用户授权 GitHub 和目标知识库后，云端自动处理新增 Star，并同步到邮件、飞书、Obsidian、Notion 或公开页面。

产品形态可以分成个人版、团队版和知识库版：从管理自己的 Star，到沉淀团队开源项目资产池。

这样它就从个人整理工具，变成面向开发者和团队的开源项目情报订阅服务。

## 第 8 页｜商业二：MCP / Skills 网络

> 建议时间：3:05-3:35

第二个方向，是接入 AI 编程工作流。

当项目档案积累起来后，可以做成 Star-DART MCP。Codex 或 Claude Code 开始新项目时，先查用户自己的开源项目库，优先复用过去收藏过的好实现。

项目档案还可以沉淀成 Skills 或 Agent 构建参考，把优秀开源项目里的工程结构和工作流变成可复用能力。

更长期看，多用户 Star 行为可以形成一个“用脚投票”的优质开源项目网络，提升开源项目检索质量。

## 第 9 页｜演示：45秒只看关键闭环

> 建议时间：3:35-4:20

这里播放 45 秒演示视频，只看闭环是否真实跑通。

第一步，用户收藏一个新 GitHub 仓库。第二步，小浣熊定时任务运行脚本，发现它还没有进入飞书 Base。

第三步，小浣熊生成项目档案，提炼定位、亮点、结构、使用建议和风险提示。

第四步，飞书里能看到最终沉淀：文档已创建，Base 已写入，仪表盘能统计分类和复用等级，周 PPT 也在同一目录下。

## 第 10 页｜总结：4句话收束

> 建议时间：4:20-4:45｜4:45-5:00 缓冲

最后用四句话总结。

第一，场景真实：GitHub Star 收藏后遗忘、难复用，是开发者和 AI 编程用户的高频痛点。

第二，闭环完整：它把定时任务、Python 脚本、小浣熊 Skill、lark-cli、飞书文档、Base、仪表盘和 Slides 串成了完整链路。

第三，当前可演示：已经有 Skill 仓库、脚本、模板、飞书 Base、仪表盘、定时任务 Prompt 和 45 秒演示视频。

第四，商业可扩展：后续可以走云端订阅、MCP / Skills 和优质开源项目知识网络。Star-DART OPC 的价值，不是多收藏一个项目，而是让每一次收藏都变成未来可以找回、判断和复用的开源资产。谢谢各位评委老师。
