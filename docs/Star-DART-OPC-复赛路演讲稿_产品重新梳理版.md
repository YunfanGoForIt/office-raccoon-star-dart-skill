## 第 1 页｜项目一句话

各位评委老师好，我的作品叫 **Star-DART OPC**。

一句话介绍，它是一个把 GitHub Star 变成开源项目知识资产的办公小浣熊 Skill。

我们平时看到一个不错的开源项目，最自然的动作就是点一下 Star。但很多时候，Star 只意味着“我当时觉得它不错”，并不意味着未来真的能找回、看懂和复用。

Star-DART 想做的事情很简单：用户仍然只做一个很轻的动作，也就是收藏仓库；后面由办公小浣熊定时发现新增 Star，生成项目文档，写入飞书多维表格，形成仪表盘，并且每周生成一份回顾 PPT。

所以这个项目的核心口号是：**现在看懂，未来找回，定期复盘。**

---

## 第 2 页｜真实痛点

这个场景来自一个非常真实的开发者习惯：我们会收藏很多项目，但真正做项目的时候，还是经常重新搜索、重新判断，甚至重新造轮子。

问题不在于我们没有收藏，而在于收藏的信息密度太低。

一个 GitHub Star 只保存了一个链接，但没有保存当时为什么收藏、这个项目适合什么场景、技术亮点在哪里、未来能不能转成自己的工具链或项目参考。

对 AI 编程用户来说，这个问题会更明显。我们让 Codex 或 Claude Code 开始做一个项目时，如果它不知道我过去收藏过哪些优秀实现，它就很可能绕过这些高质量参考，重新从通用搜索或模型记忆里找答案。

所以 Star-DART 解决的不是“如何收藏更多项目”，而是让已经收藏的项目真正成为可复用的技术资产。

---

## 第 3 页｜产品亮点

Star-DART 的核心亮点，是把一个低成本动作变成一条知识生产链。

原来用户只是得到一个 GitHub 收藏链接。现在，每个新增 Star 会被办公小浣熊处理成一份结构化项目文档，里面包括项目定位、核心亮点、技术结构、适用场景和复用建议。

原来这些项目只存在于个人短期记忆里。现在，它们会进入飞书多维表格，变成一个可以管理的资产台账，里面有技术方向、推荐动作、复用等级和文档链接。

原来我们可能看完就忘。现在，飞书仪表盘可以持续统计收藏方向，每周还会生成一份回顾 PPT，帮助用户重新看见这一周收藏了什么、哪些值得继续研究、下一步怎么用。

所以它不是一个单纯的摘要工具，而是“Star → 文档 → 台账 → 仪表盘 → 周 PPT”的完整办公链路。

---

## 第 4 页｜整体架构

接下来讲技术方案。

Star-DART 是基于办公小浣熊实现的，办公小浣熊在这里不是只负责回答问题，而是承担了计划执行、本地命令、资料研判和飞书办公交付这几个角色。

整体流程是这样的：

第一步，办公小浣熊通过定时任务周期运行。这个项目不依赖 systemd、launchd 这类系统级服务，而是直接使用小浣熊自己的定时任务能力。

第二步，小浣熊运行本地 Python 脚本 `scripts/check_new_stars.py`。这个脚本会调用 GitHub Star API，获取最近收藏的仓库。

第三步，脚本会读取飞书 Base 中已有的仓库名称和 GitHub URL，判断哪些仓库已经入库，哪些是新增 Star。

第四步，对于新增仓库，小浣熊继续按 Skill 里的规则生成项目文档，并通过 `lark-cli` 写入飞书文档、目录页、多维表格、仪表盘和 Slides。

也就是说，脚本负责稳定地“发现新增”，办公小浣熊负责把新增项目加工成真正的办公资产。

---

## 第 5 页｜技术亮点

这个项目的技术亮点主要有三点。

第一，飞书云文档承载知识资产。

Star-DART 不把结果散落成本地文件，而是把项目文档、多维表格、仪表盘和每周 PPT 都沉淀在飞书云文档体系里。这样它天然支持多端查看，也方便未来扩展到团队协作。

第二，它不是一个单点脚本，而是办公小浣熊能力组合。

这里用到了小浣熊的定时任务能力，用来定期执行；用到了本地命令能力，用来运行 Python 脚本；用到了文档生成能力，用统一模板生成项目档案；还通过飞书插件和 `lark-cli` 完成文档创建、Base 写入、目录维护和仪表盘更新；最后，每周再用 PPT 生成能力做一次记忆强化。

第三，脚本和 Agent 的分工比较清晰。

脚本只做确定性强的事情，比如抓取 Star、读取 Base、比对去重。Agent 做更需要理解和判断的事情，比如判断项目价值、生成结构化文档、给出推荐动作。这种分工让流程更稳定，也保留了 AI Agent 的判断能力。

---

## 第 6 页｜实现过程与当前闭环

当前这套 Skill 已经收敛成一个比较清晰的闭环。

第一步是定时任务。办公小浣熊按周期执行脚本，默认 3 小时轮询一次；演示时可以临时缩短周期，便于看到效果。

第二步是本地命令。小浣熊运行 `python3 scripts/check_new_stars.py`，脚本完成 GitHub Star 抓取，并和飞书 Base 做去重。

第三步是文档生成。对每个新增仓库，小浣熊参考 `references/doc_template.md` 生成项目档案，不是简单翻译 README，而是提炼这个项目为什么值得收藏、适合怎么复用。

第四步是飞书插件连接。小浣熊通过 `lark-cli` 创建飞书子文档，把文档入口追加到目录页，并写入飞书 Base，维护仪表盘。

第五步是 PPT 生成。每周把这一周新增 Star 汇总成回顾 PPT，帮助用户强化记忆。

从仓库资产来看，现在已经有 `SKILL.md`、核心脚本、项目文档模板、周回顾 PPT 模板、定时任务 Prompt，以及飞书目录页、Base、仪表盘和 PPT 这些最终办公产物。

---

## 第 7 页｜效率价值与潜在用户

Star-DART 的效率价值，主要来自减少重复整理。

手工做这件事的时候，用户要打开 GitHub Star 列表逐个检查；看到项目以后重新读 README；读完以后再写笔记；然后手动维护 Excel 或表格；最后还要靠自己记得这一周到底看过什么。

Star-DART 把这些动作拆开交给小浣熊处理。

定时脚本只输出新增仓库；项目文档用统一模板生成；资产台账用飞书 Base 维护；统计图交给 Base 仪表盘；每周回顾交给 PPT。

这样一来，用户真正需要做的不是重复搬运信息，而是复核小浣熊的判断，决定哪些项目值得继续研究、试跑或沉淀成自己的工具。

潜在用户也比较明确：AI 编程重度用户、独立开发者、学生和科研新人、技术团队，以及做技术内容的人。越依赖开源项目和 AI 编程，越需要一个自己的开源项目知识资产库。

## 第 8 页｜商业拓展一：云端订阅服务

当前版本验证的是个人可运行的最小闭环。往后看，第一个商业拓展方向是云端订阅服务。

现在的 Skill 模式适合复赛演示和个人使用，但它需要用户自己配置 GitHub、飞书 token，也需要有办公小浣熊或本地环境定时执行。

如果做成 Star-DART 云服务，用户只需要授权 GitHub 和目标知识库，云端就可以自动处理新增 Star，并把结果推送到邮件、飞书、个人知识库，甚至同步到 CDN。

这个方向可以有三个产品形态。

个人版，帮助开发者订阅自己的 GitHub Star，每次新增自动生成项目档案。

团队版，汇总团队成员关注的项目，形成团队开源项目资产池。

知识库版，在云端构建专属项目知识库，支持同步到飞书或公开页面。

这样，Star-DART 就不只是一个个人整理工具，而可以变成一个面向开发者和团队的开源项目情报订阅服务。

---

## 第 9 页｜商业拓展二：MCP / Skills 与优质开源项目网络

第二个拓展方向，是把 Star-DART 进一步接入 AI 编程工作流。

当用户积累了足够多项目档案之后，这些档案就可以成为 Codex、Claude Code 这类 AI 编程工具开工前的检索来源。

比如做一个 Star-DART MCP。每次开始一个新项目时，AI Agent 可以先查询用户自己的开源项目知识库，看看过去收藏过哪些可复用项目，避免重复造轮子。

再进一步，项目档案可以沉淀成 Skills 或 Agent 构建参考，把优秀开源项目里的设计方式、工程结构和工作流变成可复用能力。

如果用户规模继续扩大，还可以把多用户的 Star 行为汇总起来，形成一个“用脚投票”的优质开源项目网络。

这能解决一个更大的问题：GitHub 搜索对模糊需求并不友好，而 AI 搜索推荐的项目质量也不稳定。Star-DART 可以用真实用户收藏行为和结构化项目档案，构建更高质量的开源项目检索入口。

## 第 10 页｜录制视频演示

接下来是作品演示部分。

现场我会播放一段大约 3 分钟的录制视频，稳定展示完整闭环。

视频里会先展示用户收藏一个 GitHub 仓库。然后办公小浣熊的定时任务运行本地脚本，发现这个仓库还没有出现在飞书 Base 中。

接着，小浣熊会为这个新增仓库生成项目文档，提炼项目定位、核心亮点、技术结构、使用建议和风险提示。

最后，视频会展示飞书里的沉淀结果：项目文档已经创建，Base 台账已经写入，仪表盘可以看到分类和复用等级统计，每周回顾 PPT 也会沉淀在同一个飞书目录下。

这个演示想证明的重点是：办公小浣熊不是完成一次问答，而是在执行一套可复用、可持续运行的办公自动化 Skill。

---

## 第 11 页｜总结：对齐复赛评分

最后总结一下 Star-DART OPC 为什么适合这个复赛赛题。

在场景洞察上，它解决的是 GitHub Star 收藏后遗忘、难复用的问题，这是开发者、学生、科研用户和 AI 编程用户都会遇到的真实高频痛点。

在技术方案上，它不是一个单脚本 Demo，而是把办公小浣熊定时任务、Python 脚本、`lark-cli`、飞书文档、Base、仪表盘和 Slides 串成了一个完整闭环。

在当前成果上，它已经有 Skill 仓库、样例、测试、飞书 Base、仪表盘、定时任务 Prompt 和录制视频演示路径。

在商业价值上，它先从个人办公闭环切入，后续可以拓展成云端订阅服务、MCP / Skills，以及优质开源项目知识网络。

所以 Star-DART OPC 的价值，不是让人多收藏一个项目，而是让每一次收藏都变成未来可以找回、判断和复用的开源资产。
